# Data Cube Tracking — Roadmap

เอกสารนี้สรุปแนวทางพัฒนาระบบสำหรับใช้คุยและจัดลำดับงานร่วมกันในทีม

## สถานะปัจจุบัน

- ระบบทำงานบน Docker Compose พร้อม PostgreSQL
- เปิดทดสอบได้ผ่าน `http://localhost:3000` และ Public IP ของเครื่อง
- มีผู้ใช้ `ray` ในบทบาท Admin แล้ว
- ระบบแจ้งเตือนในแอปมีการตั้งค่าได้ที่หน้า Team สำหรับ Admin
- แจ้งผู้เกี่ยวข้องเมื่อได้รับมอบหมายหรือมีการเปลี่ยนสถานะงาน
- มีตัวตรวจ SLA อัตโนมัติทุกชั่วโมง: เตือนก่อนครบกำหนด, เตือนงานเกินกำหนด และ escalation ถึง Admin

> ก่อนใช้งานจริงภายนอกองค์กร ควรเพิ่ม Domain, HTTPS และแผนสำรองฐานข้อมูล

## เป้าหมาย

ทำให้ Data Cube Tracking เป็นศูนย์กลางติดตามงานภายในบริษัท: ทีมแจ้งและจัดการงานได้ง่าย, ผู้รับผิดชอบได้รับการเตือนตรงเวลา, และผู้บริหารเห็นสถานะงานได้ชัดเจน

## แผนงาน

### 0. Foundation และการดูแลระบบ — เสร็จแล้ว (ระยะทดสอบ)

- Docker Compose: Web app, PostgreSQL และ SLA scheduler
- Persistent volume สำหรับฐานข้อมูลและไฟล์อัปโหลด
- Public IP สำหรับทดสอบภายในทีม

งานก่อนเปิดใช้งานจริง

- จด Domain และตั้งค่า HTTPS ผ่าน reverse proxy
- ตั้ง automated backup และทดสอบการกู้คืน PostgreSQL
- เปลี่ยนรหัสผ่านชั่วคราวของบัญชีทดสอบ และกำหนดนโยบายรหัสผ่าน
- จำกัด firewall/การเข้าถึงฐานข้อมูลให้เหมาะสม
- เพิ่ม monitoring ของ service, disk space และ backup

### 1. Admin Settings และสถานะงานแบบตั้งค่าได้ — งานถัดไป

เป้าหมาย: แยกการตั้งค่าระบบออกจากหน้า Team และให้ Admin ปรับ workflow ได้โดยไม่แก้โค้ด

- สร้างหน้า `/settings` และซ่อนเมนู/ปฏิเสธการเข้าถึงสำหรับผู้ที่ไม่ใช่ Admin
- ย้าย Notification/SLA settings จากหน้า Team ไปหน้า Settings
- ให้ Admin เพิ่ม, แก้ไข, เรียงลำดับ, เปิด/ปิดสถานะงาน และกำหนดสีได้
- เพิ่มสถานะเริ่มต้น `Pending` และ `Waiting BU`
- กำหนดชนิดสถานะ: active, paused และ closed เพื่อควบคุมการนับ SLA
- ตั้ง `Waiting BU` เป็น paused SLA: ไม่แสดง/ไม่แจ้ง Due Date และไม่เกิด overdue ระหว่างรอ
- บันทึก `waitingBUStartedAt` เมื่อย้ายงานเข้า `Waiting BU` พร้อมเก็บใน Activity Log ว่าใครเป็นผู้เปลี่ยนสถานะ
- เมื่อย้ายออกจาก `Waiting BU` ให้กลับไปใช้ SLA/Due Date ตามกติกาที่ Admin เลือก (กำหนด due date ใหม่หรือคืนค่าเดิม)
- แจ้งเตือนผู้รับผิดชอบติดตาม BU ทุก 7 วัน โดยตั้งรอบและเปิด/ปิดได้จาก Settings
- แสดงจำนวนวันที่รอ BU บน ticket และ dashboard เพื่อให้ติดตามงานค้างจาก BU ได้
- คงสถานะเดิมและข้อมูล ticket เดิมไว้ทั้งหมด

### 2. Google Workspace SSO และผู้ใช้ภายในบริษัท

เป้าหมาย: ให้พนักงานเข้าใช้ด้วยบัญชี Google Workspace เท่านั้น ลดภาระการสร้างและดูแลรหัสผ่าน

- Login ด้วย Google OAuth / Google Workspace
- จำกัดอีเมลตามโดเมนบริษัท
- กำหนดบทบาท Admin, Manager และ Member
- ผูกผู้ใช้เดิมกับบัญชี Google โดยไม่ทำให้งานเดิมหาย
- กำหนดขั้นตอน offboarding เมื่อพนักงานออก

ข้อสังเกต: หากบริษัทมี Google Workspace อยู่แล้ว โดยทั่วไปไม่มีค่า license เพิ่มสำหรับ SSO ของระบบนี้ แต่ต้องตั้งค่า OAuth consent screen และสิทธิ์ใน Google Cloud Project

### 3. Notification และ SLA — เสร็จแล้ว (เวอร์ชันแรก)

- Admin ตั้งเปิด/ปิดประเภทการแจ้งเตือนได้จากหน้า Team
- แจ้งผู้ที่เกี่ยวข้องกับ ticket ก่อน: ผู้รับผิดชอบ, ผู้สร้างงาน และผู้ที่ถูกกล่าวถึงตามเหตุการณ์
- ตั้งช่วงเวลาเตือนก่อนครบกำหนดได้
- แจ้ง overdue และ escalate ถึง Admin หลังเกินกำหนดตามเวลาที่ตั้งไว้
- Scheduler ทำงานทุกชั่วโมง พร้อมป้องกันการแจ้งเตือนซ้ำในวันเดียวกัน

งานต่อยอดที่ควรพิจารณา

- เพิ่มช่องทาง Email และ Google Chat/Slack
- ตั้ง escalation ตามทีม, Business Unit หรือประเภทงาน
- ตั้ง quiet hours และสรุปแจ้งเตือนรายวัน
- เพิ่มประวัติการส่ง notification เพื่อ audit

### 4. Executive Dashboard และรายงาน

เป้าหมาย: ให้ผู้บริหารเห็นภาพงานค้างและความเสี่ยงได้ทันที

- จำนวนงานตามสถานะ, ทีม และ Business Unit
- งานใกล้ครบกำหนด/เกินกำหนด และแนวโน้ม SLA
- ภาระงานของแต่ละทีมและผู้รับผิดชอบ
- Export รายงาน CSV/PDF และรายงานสรุปรายสัปดาห์

### 5. Internal Request Form

เป้าหมาย: รับคำขอจากคนในบริษัทอย่างเป็นมาตรฐาน โดยผู้ส่งต้องผ่านการยืนยันตัวตน

- Form builder สำหรับ Admin ตั้งฟิลด์และประเภทคำขอเองได้
- จำกัดเฉพาะผู้ใช้บริษัทผ่าน SSO
- แปลงคำขอเป็น ticket พร้อมทีม/Business Unit/ผู้รับผิดชอบเริ่มต้น
- ติดตามสถานะคำขอและแจ้งผู้ร้องขอ

### 6. MCP สำหรับ AI แบบผูกตัวตนผู้ใช้

เป้าหมาย: ให้ Claude, Codex หรือ AI client อื่นเชื่อมกับระบบได้อย่างปลอดภัย และเห็น/จัดการได้เฉพาะงานตามสิทธิ์ของผู้ใช้

- MCP server ผ่าน HTTPS พร้อม OAuth/Google Workspace identity
- Tools แบบ read-only ก่อน: ดูงานของฉัน, งานใกล้ครบกำหนด, สรุปสถานะ
- Tools แบบเขียนต้องยืนยัน: สร้าง ticket, เปลี่ยนสถานะ, เพิ่ม comment, มอบหมายงาน
- ตรวจสิทธิ์ทุก request ตาม user และ role ฝั่ง server
- Audit log: AI ใด, ผู้ใช้ใด, ทำคำสั่งใด, เมื่อไร
- Rate limit และ approval policy สำหรับคำสั่งที่มีผลกระทบ

### 7. AI Manager

เป้าหมาย: ให้ AI ช่วยติดตามงานโดยไม่แทนที่การตัดสินใจของทีม

- สรุปงานค้าง/เสี่ยงตามแต่ละผู้ใช้หรือทีม
- เตือนงานที่ไม่มีความคืบหน้า
- เสนอผู้รับผิดชอบและลำดับความสำคัญ
- สร้าง weekly digest ให้หัวหน้าทีม
- คำสั่งเปลี่ยนแปลงข้อมูลต้องให้ผู้ใช้ยืนยันก่อนเสมอ

## ลำดับที่แนะนำ

1. Admin Settings และสถานะงานแบบตั้งค่าได้
2. Google Workspace SSO และจำกัดเฉพาะผู้ใช้บริษัท
3. Backup, HTTPS และ monitoring ก่อนเปิดใช้งานจริง
4. Executive Dashboard
5. Internal Request Form
6. MCP แบบ read-only และ user-scoped
7. AI Manager และการจัดการงานแบบต้องยืนยัน

## ประเด็นสำหรับคุยกับหัวหน้า

- ใครเป็นเจ้าของระบบและเป็น Admin หลัก
- โดเมน Google Workspace ที่อนุญาตให้เข้าใช้
- ทีม/Business Unit และกติกา SLA ที่ต้องการ
- ช่องทางแจ้งเตือนที่ต้องใช้จริง: In-app, Email, Google Chat หรือ Slack
- ขอบเขตที่อนุญาตให้ AI อ่านข้อมูลและดำเนินการแทนผู้ใช้
- ผู้รับผิดชอบ backup, HTTPS, monitoring และค่าใช้จ่าย infrastructure
